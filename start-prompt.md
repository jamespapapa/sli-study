이 레포는 웹 전용 프론트엔드/백엔드 프로젝트를 5개의 서브에이전트로 운영하는 라운드 기반 오케스트레이션 예시다.
항상 `planner -> designer -> front-developer + back-developer(병렬) -> tester` 순서로 수행하라.

라운드마다 아래 순서를 지킨다.

1. `./scripts/init_round.sh {ROUND_NO}` 로 해당 라운드 핸드오프 파일을 준비한다.
2. 각 단계 에이전트는 `AGENTS.md` 의 책임과 규칙에 맞춰 작업한다.
3. 단계별 산출물은 반드시 `handoffs/round_{ROUND_NO}/` 아래의 지정된 Markdown 파일로 남긴다.
4. 라운드 종료 시 `./scripts/validate_round.sh {ROUND_NO}` 로 문서 완결성을 검증한다.

이 프로젝트는 웹 전용 프론트엔드/백엔드 구조를 전제로 한다.
프론트엔드는 `frontend/`, 백엔드는 `backend/` 에서 작업하며, 각 단계는 자신이 실행한 `lint`, `test`, `compile`, `build` 등의 결과를 핸드오프에 남긴다.

구현 대상 예시는 다음과 같다.
한국인 디자이너 홈페이지. 멋진 디자이너의 포트폴리오가 전시된 홈페이지. 가장 하단에는 문의하기 폼이 존재하며 문의 내용 저장이 가능해야 한다.

종료 조건은 아래와 같다.

1. 마지막 라운드의 `tester_report.md` 상태가 `PASS` 이다.
2. 해당 라운드의 필수 핸드오프 문서가 모두 존재하고 검증 스크립트를 통과한다.
3. 프론트엔드와 백엔드의 필수 검증 명령 결과가 모두 성공으로 기록되어 있다.

계속 진행 조건은 아래와 같다.

1. `tester_report.md` 상태가 `REVISE` 이다.
2. 필수 핸드오프 문서가 누락되었거나 placeholder가 남아 있다.
3. 실행 가능한 검증 명령 중 실패가 있다.

중단 조건은 아래와 같다.

1. `tester_report.md` 상태가 `BLOCKED` 이다.
2. 동일한 blocker가 2개 라운드 연속 반복되고, 다음 라운드에서 달라지는 실행 계획이 없다.

총점은 라운드 추세 파악용 참고치로만 사용한다. 총점이 올랐는지 여부만으로 종료하지 않는다.
